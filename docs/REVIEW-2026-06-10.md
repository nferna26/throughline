# Field-test review — 2026-06-10

**Scope.** This is the v0.4.0 field-test review. The operator used the signed build as a
paying reader on a real Mac — bought the $20 allowance, discovered and imported books,
read, asked the tutor — and filed six complaints. Six investigation agents ran them down:
**all six complaints validated**, every one with a code-level root cause. A fresh
stranger's-walk over the same build found more.

**Method.** Code-level investigation of the app repo and the relay repo
(`~/code/throughline-ai-proxy`); HEAD on `main` is identical to tag v0.4.0 for every cited
file, so all line numbers are the shipped build. Live checks where safe: the relay is up
and answering (`/health` ok, unauthenticated 401), the updater endpoint 404s, and
**Gutendex was down for the entire test window** (TLS accepted, zero bytes returned, three
attempts) — the condition behind the catalogue complaint. Existing suites pass as-is
(vitest 220/220, `cargo test` targeted suites green, proxy vitest 19/19) — every Red below
is a new failing test. Find-only — **nothing has been fixed yet**. (`docs/AUDIT.md`, which
CLAUDE.md asks reviewers to reconcile, still does not exist; this report reconciles
against `docs/REVIEW-2026-06-09.md` where the walks overlap.)

Severity: **FT-P1** = broken on a paying reader's Mac. **FT-P2** = a reader would feel
it. **FT-P3** = polish.

## FT-P1 — broken on a paying reader's Mac

### FT-1. Section briefing always 400s in company mode — the app sends max_tokens 768, the relay gate is 700

Every company-mode Section briefing is deterministically rejected before reaching the
model: the app attaches `SECTION_BRIEFING_MAX_TOKENS: u32 = 768`
(`src-tauri/src/commands/ai.rs:311`, selected at :325, wired at :477), while the relay's
shape-lock rejects `max_tokens > MAX_OUTPUT_TOKENS` with "max_tokens too large"
(`throughline-ai-proxy/src/shape.ts:36-37`, configured `"700"` at
`throughline-ai-proxy/wrangler.toml:13`). Only SectionBriefing breaks — brief=200,
deep=450, durable_note=256, prepare_next=512 (ai.rs:303-307) all pass the gate. Root
cause: cross-repo contract drift — commit 35cf3ee raised the app ceiling for headroom, the
proxy was provisioned at 700 in eb8b319 and never moved, and no test in either repo pins
app ceilings ≤ relay gate. This finding spans both repos.

🔴 **Red — failing test first.** In the `commands/ai.rs` tests module (next to the
existing depth-ceiling test at :1293-1335): `every_mode_ceiling_fits_the_relay_contract()`
— assert for every (StubMode, Depth) pair that `max_tokens_for(mode, depth) <=
COMPANY_RELAY_MAX_OUTPUT_TOKENS`, where the new `pub const
COMPANY_RELAY_MAX_OUTPUT_TOKENS: u32 = 700` documents the deployed shape gate (mirror of
the proxy's wrangler.toml `MAX_OUTPUT_TOKENS`). Fails today: SectionBriefing resolves to
768 > 700. Companion integration red in `src-tauri/tests/company_proxy_mock.rs` using the
existing `mock_proxy` helper: a company call against a mock that (like shape.ts) replies
400 above 700 must instead capture a request body with `max_tokens` ≤ 700. Hermetic —
loopback mock, no live relay.

🟢 **Green — smallest change to pass.** Clamp at the company call boundary in
`cmd_ai_ask` (ai.rs:477): `let mt = max_tokens_for(stub_mode, answer_depth); let mt = if
matches!(provider, settings::AiProvider::Company) { mt.min(COMPANY_RELAY_MAX_OUTPUT_TOKENS) }
else { mt };` — one constant + one `min()`. BYO providers keep the 768 headroom.
Separately, server-side: raising wrangler.toml `MAX_OUTPUT_TOKENS` to e.g. 800 is a
config-only proxy deploy that un-breaks shipped v0.4.0 readers without an app release —
but the app-side clamp is what makes the contract testable.

**Done when:** `cargo test` passes including the new contract test; the mock-proxy test
captures `"max_tokens":700` (not 768) for a company section_briefing call; manual
golden-loop check: a company reader's Section briefing streams all five parts instead of
erroring.

### FT-9. Reading a section to its end almost never counts as finishing it

Completion-by-scroll asks whether the paragraph at the **top** of the viewport has
entered the last 5% of the section's characters (`src/screens/TextReader.tsx:314`,
topOffset from handleScroll :264-286). At maximum scroll the top-visible paragraph sits
about one viewport before the end, so the rule needs a section ≥ ~20 viewport-heights to
ever fire; a section that fits one screen never fires handleScroll at all; and a bigger
window makes finishing harder. Downstream, completion drives Today's assignment and pace
(books.rs:210-216, plan.rs:229-240, :127-180), and book completion requires every section
completed (sessions.rs:204-227) while goNext clamps at the last section
(TextReader.tsx:295-297) — so a book whose last section is short can never reach "You
finished". Readers who press Next past each chapter get credit (the visited-and-passed
rule, :309-311); readers who stop at the chapter's end and press Finish — the primary CTA
— are told "0 sections done", then get the recovery panel for being "Behind" while
reading every day. No test pins the contents of completedSectionIds.

🔴 **Red — failing test first.** Two layers. (1) New `src/sectionCompletion.test.ts` for
a new pure helper: `endReached({scrollTop:3200, clientHeight:800, scrollHeight:4000}) ===
true` (bottom within ~32px slack); `endReached({scrollTop:0, clientHeight:800,
scrollHeight:700}) === true` (section fits without scrolling); `endReached({scrollTop:0,
clientHeight:800, scrollHeight:4000}) === false`. Red: the module doesn't exist. (2) In
TextReader.test.tsx: render a single-section book, `Object.defineProperty` the
`.tl-reader-main` container's scrollTop=3200/clientHeight=800/scrollHeight=4000,
fireEvent.scroll, click Finish→Finish, assert `cmd_end_session` called with
completedSectionIds containing 's1'. Fails today: jsdom paragraph offsetTop=0 →

🟢 **Green — smallest change to pass.** Add `src/sectionCompletion.ts` with the pure
`endReached()` predicate; in TextReader keep a `reachedEnd: Set<sectionId>` updated in
handleScroll from container geometry (and once after text paints, for sections shorter

**Done when:** both new tests green; `grep -n '0.95' src/screens/TextReader.tsx` returns

## Verdict

**All six complaints validated** — none was a misunderstanding, user error, or a transient
glitch. As filed and as found:

1. **"Section briefings failed, then the tutor said it was paused"** — real and
   deterministic: a cross-repo contract drift (FT-1) fed a breaker that can't tell a
   client error from an outage (FT-2), behind a sheet that lies about what was sent
   (FT-3), with raw JSON in the margin (FT-10) and silent auto-refires multiplying the
   damage (FT-13).
2. **"The gauge read ~75% minutes after I paid; the history is full of orange internal
   rows"** — the gauge is a hardcoded decoration (FT-11; the server-side metering itself
   verified honest — the failed briefings cost $0.00), and the history rows leak a mode
   id and a hostname in alarm orange (FT-12).
3. **"The window can't be dragged, the icon isn't ours, updates error out"** — a missing
   one-line capability (FT-4), the stock scaffold icon (FT-14), and an updater that has
   never once worked because all four releases are unpublished drafts (FT-5, FT-15).
4. **"Before you read is redundant mush"** — redundant by design (FT-16), with
   front-matter leaking into the excerpt (FT-17) and a genre-blind prompt (FT-18); the
   surrounding Today card triple-states "plan ready" (FT-19) and opens a guilt ledger on
   day one (FT-20, FT-34).
5. **"Settings is broken and full of internals"** — a real CSS layout bug every paying
   reader sees (FT-21), a toggle that flatly lies in the paid build (FT-6), hostnames and
   legalese at the wrong altitude (FT-22, FT-36, FT-35), warning chrome on normal states
   (FT-23), and no editorial hierarchy (FT-24).
6. **"Search only finds a tiny slice of the library"** — Gutendex was down the entire
   window, and the app turned that outage into confident lies: a 200-title seed presented
   as the whole library with absences asserted as fact (FT-7), a cry-wolf offline pill
   (FT-25), an impatient click that cancels the live upgrade (FT-26), silent truncation
   (FT-27), invisible filters (FT-28), and a never-shown catalogue count (FT-37).

The stranger's-walk findings (FT-8, FT-9, FT-29 through FT-33, FT-38, FT-39) are the
sharper lesson, because nobody complained about them yet: errors delivered to a dialog
API the platform never implemented, and a completion rule that is geometrically
unreachable, were both sitting under a fully green suite.

That is what this field test proves about the gap between suite-green and reader-ready.
The 2026-06-09 review ran every test, read every file, and called the architecture
trustworthy — and it is. But the suite pins what the code does, not what a Mac does with
